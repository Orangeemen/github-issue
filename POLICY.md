山东大学开源软件镜像站致力于为技术爱好者、工程师、科研人员等开源爱好者提供优质的开源软件镜像服务。我们欢迎不同专业、不同背景的组织或个人参与到镜像站的建设当中。所有参与者（申请人、维护者、贡献者）均应提前认真阅读以下规范与说明。

## 一、行为规范

本镜像站致力于营造开放、专业、高效的协作环境：

### ✅ 我们鼓励
- 基于事实与解决方案的技术讨论
- 提供清晰的复现步骤、日志与环境信息
- 对维护决策提出建设性反馈与替代方案
- 尊重他人时间与资源，按需精简请求范围

### 🚫 禁止行为
- 人身攻击、歧视性言论、情绪化指责
- 恶意提交重复/虚假/无意义申请
- 滥用自动化脚本高频请求，影响服务稳定性
- 在公开渠道泄露私有配置或敏感信息

如果出现上述不当行为，根据情况的严重性，管理员有权进行适当处理，并在必要时向相关机构或组织报告。

## 二、收录原则

为了合理利用镜像站的存储与带宽资源，确保持续、稳定地为开发者提供服务，我们对申请收录的开源项目制定了以下原则：

1. **合法合规与开源**：项目不得包含违规内容，并且必须基于公认的开源协议，不涉及法律争议或侵权风险。
2. **上游稳定性**：上游源应处于积极维护状态。已停止维护且无重大存档价值的项目一般不予考虑。
3. **技术可行性**：必须支持可靠、主流的同步方式（如 rsync、HTTP/HTTPS），且能够正常进行稳定、高效的增量同步。
4. **资源合理性**：同步造成的带宽与存储消耗应在镜像站承载范围内，且同步频率需合理。
5. **实际需求**：优先考虑有切实需求、应用广泛、受众群体大的项目。

*注：由于镜像站资源限制，我们无法保证所有符合规则的镜像申请都能被接受，敬请谅解。*

### 出现以下任一情况，原则上不予镜像收录：

1. 违反相关法律法规，或由于版权、许可证等因素不允许再分发的。
2. 缺乏合适且高效的增量同步方案的。
3. 数据体量巨大难以进行高效同步的。
4. 属于闭源商业软件或项目的。
5. 受众群体极其有限的。
6. 已存在更优获取渠道的。

## 三、GitHub Release 专属收录细则

为了合理利用镜像站的存储与带宽资源，确保持续、稳定地为开发者提供服务，我们对申请收录的开源项目制定了以下偏好与底线规则：

### 1. 基本原则与收录偏好

1. 合法合规与开源：项目不得包含违规内容，且必须基于公认的开源协议。
2. 推崇“小而美”：我们偏好于那些体积小巧、功能聚焦、代码精简的优秀开源项目。
3. 偏好轻量级应用：倾向于收录非 Electron 架构的项目；倾向于收录不自带庞大运行时（Runtime）的产物（例如：依赖系统已安装框架的 C# / .NET 程序，而非自包含（Self-contained）的巨无霸包）

### 将会慎重考虑收录的情况

1. 臃肿的发行版：如果该项目的 Release 默认打包了整个运行环境（如 Electron、大型虚拟机池），本站将谨慎收录或直接拒绝该类产物的全量镜像申请。
2. 交叉编译的枚举泛滥：对于在 Release 中枚举列出数十种不同架构（如 x86, x64, arm, arm64, mips64, riscv64 等各个系统组合）的项目，我们原则上只收录主流平台（如 Win/Mac/Linux 的 x64/arm64 核心产物），不收录全量文件，申请者需在 Issue 中明确指出需要加速的 1~3 个核心基础文件。
3. 高度重复的产物：如果项目的 Release 同时且大量提供 Installer（安装版）和 Portable（便携版/绿色版），且内容高度重复，本站将拒绝全量收录，申请者需明确指出需要加速的少数核心基础文件。

## 四、标签说明

### 状态标签

- 🟡 `pending`：待审核（新提交默认）
- 🔍 `evaluating`：评估中（维护者正在调研）
- 👍 `accepted`：已接受，等待配置
- 🚧 `setting`: 正在配置与测试
- ✅ `deployed`：已上线，可访问
- ❌ `rejected`：未通过，附原因说明
- ⏸️ `on-hold`：暂缓（资源/技术限制，可后续重启）
- 🗑️ `invalid`：无效申请（重复/信息不全/不符合规则）

### 优先级标签（可选）

- 🔴 `high-priority`：高需求 / 紧急修复
- 🟡 `medium`：常规需求
- 🟢 `low`：低优先级
